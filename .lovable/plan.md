# Loans, Bills, Budgets, Savings, Investments, Reports & PWA Polish Plan

You raised 8 distinct concerns. Here's how each will be solved — minimum changes, maximum impact, no schema bloat.

---

## 1. Remove "loan disbursement to account" (your priority concern)

**Today:** Adding a loan creates an `income`-tagged `loan-disbursement` transaction on a chosen asset account and bumps that account's balance. Even though it's tag-filtered from reports, it still shows up in the transaction list and inflates the account, which feels redundant.

**Fix:**
- `LoanForm.tsx`: drop the `linked_account_id` (disbursement target) field entirely. The loan is just a liability balance — period.
- No transaction is created on disbursement.
- `linked_account_id` (in DB) is **only** used for repayment source — re-purpose the field, ask for it inside `LoanPaymentForm` instead (so users can pick where each repayment is paid from at payment time).
- Net worth still works correctly: liability ↑ on disbursement, assets unchanged (you decide what to do with the borrowed cash via your normal income/transfer flow if you actually receive it).

**Result:** zero report pollution, zero phantom transactions, no double-listing.

---

## 2. Loan repayment "progress not updating" bug

**Root cause:** After payment, `LoansDashboard` re-renders but `LoanCard` reads the stale `loan` prop. `LoanDetailPanel` uses local `freshLoan` lookup but the parent's `accounts` prop only refreshes when `onRefresh()` triggers `fetchData()` in `Dashboard.tsx`. The `LoanPaymentForm` calls `onSuccess()` and dispatches `dashboard-refresh` — so refresh works — but the payment list inside `LoanDetailPanel` (`fetchPayments()`) is only called on mount, not after a payment.

**Fix:**
- After a successful payment, `LoanDetailPanel` calls `fetchPayments()` again (re-fetch payment history).
- `paidAmount` and `paidPercentage` recompute from the freshly-fetched `loan.balance` (already wired — but we'll force a re-read by passing `loan` from `accounts.find(...)` after refresh).
- Add an explicit `onPaymentSuccess` callback chain: `LoanPaymentForm.onSuccess` → `LoanDetailPanel` re-fetches payments + calls parent `onRefresh` → progress bar updates immediately.

---

## 3. Bills & Subscriptions — clean pay flow with no fraud / mismatching

**Problems today:**
- `markPaid()` blindly picks the **first active account** as the source — silently wrong.
- No confirmation step → user can mark something paid by mistake.
- No category linkage → bill payments sit uncategorized in reports.

**Fix:**
- Add a **"Pay From Account"** selector inside the Pay button (small inline confirmation popover) before recording payment.
- On pay: tag the resulting transaction with `bill-payment` and the bill's category; auto-map bill `category` to a matching expense `Category` (create if missing once, then reuse).
- Auto-pay bills (`auto_pay = true`) still confirm before deducting — never deduct silently.
- Edit/Update: existing form already supports this; we'll simply make the Edit option visible (it currently only triggers from dropdown — we'll surface a clear pencil button on each card).
- Add a small "next 30 days obligations" indicator so the user understands what's coming.

## 4. Budget templates — confirm live & wire properly

The "mock" part is only the template definitions themselves (Student / Family / Freelancer presets), which is fine because they are starting points the user customises.

**Fix:**

- After apply, immediately navigate to the Budgets list so the user sees them.
- Add an "Apply with my currency" toggle (currently hardcoded TZS) — uses the user's `default_currency` from settings.
- No mock data persists — all flows already write to DB.

**Fix sweep — small CSS-only changes:**
- Add `overflow-x-hidden` to `<main>` in Dashboard.

- Wrap currency-heavy stat values in `text-base sm:text-lg` instead of fixed `text-lg` so they shrink on narrow screens.
- Replace `whitespace-nowrap` with `break-words` on labels.
- Keep horizontal scroll **only** on the desktop tab nav (per your instruction).
